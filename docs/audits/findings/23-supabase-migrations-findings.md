## Summary

The Supabase migration set covers four files: an init schema (`accounts`, `server_profile_pointers`, `entitlements`), a Founders extension, a v2 rewrite (`app_config`, `cosmetics_catalog`, `cosmetic_unlocks`, `achievement_progress`, triggers), and a follow-up fix-grants migration. RLS is enabled on every public table and policies use `auth.uid()` correctly — no hard-coded UUIDs, no obvious permissive policies. The Edge Functions correctly use the service-role key only on the server side.

However, two **high-severity** problems exist. First, the v2 migration creates four new tables (`app_config`, `cosmetics_catalog`, `cosmetic_unlocks`, `achievement_progress`) but never issues any `GRANT` statements — this is the exact bug that the follow-up `fix_table_grants` migration was written to address for the v1 tables, yet the same fix was never applied to the v2 tables. Per the project's own migration-comment evidence, this results in `42501 permission denied` for every client read/write before RLS is even evaluated; `fetchFoundersLaunchAt()` in `PurchasesContext.tsx` actively depends on `app_config` and will silently fail. Second, all four `SECURITY DEFINER` functions (`handle_new_user`, `handle_new_account`, `grant_cosmetics_for_entitlement`, `auto_grant_new_cosmetic_to_founders`) omit an explicit `SET search_path`, leaving them susceptible to search-path-based privilege escalation per long-standing Postgres / Supabase linter guidance.

Beyond those, the audit flags one med-severity unconstrained public-read policy on `app_config`, a missing-`WITH CHECK` pattern on two UPDATE policies (relies on the USING-as-fallback rule), a Supabase performance miss where every policy uses bare `auth.uid()` instead of `(select auth.uid())`, and several low/nit items around idempotency, FK indexing, and policy-coverage documentation.

All SQL migration files are read-only per `_RULES.md` §"Absolutely forbidden". Every finding below is `proposed` — a new forward-only migration is the correct fix path.

| db-002 | high | infra/supabase/migrations/20260401000000_init.sql:47-58,132-140 ; 20260501000000_purchases_v2.sql:185-214,228-240 | All four `SECURITY DEFINER` functions omit an explicit `SET search_path`. | db-003 | med | infra/supabase/migrations/20260501000000_purchases_v2.sql:57-59 | `app_config` policy is `using (true)` — every key in this table is publicly readable by any anon or authenticated client. Today only `founders_launch_at` lives there (non-sensitive), but the table is a generic key/value store; an operator who later adds a key like `support_api_token`, `feature_flag_internal_pricing`, or any other config a TypeScript client should not see would leak it instantly. The Supabase docs explicitly warn against `using (true)` on configuration tables for this reason. | Either (a) constrain reads to a whitelist of public keys, e.g. `using (key in ('founders_launch_at'))`, and require new public keys to extend the policy; or (b) split into `public_app_config` (public read) and `private_app_config` (service_role only) and migrate the seed accordingly. Document the policy intent in the migration comment so future contributors don't put a secret in `app_config` and assume RLS protects it. | fixed |
| db-004 | med | infra/supabase/migrations/20260401000000_init.sql:29-31,92-94 | UPDATE policies on `accounts` and `server_profile_pointers` use only `USING (...)` with no `WITH CHECK (...)`. Postgres falls back to USING-as-WITH-CHECK by default, so this is not currently exploitable (the same `id = auth.uid()` predicate is applied to both the existing and the proposed new row), but the implicit fallback is a subtle footgun: a future maintainer who adds a column-level condition to USING (e.g. `using (id = auth.uid() AND deleted_at IS NULL)`) will not realize the same predicate now governs writes, potentially allowing a user to "rescue" a soft-deleted row by updating it. The `server_profile_pointers` INSERT policy already uses an explicit `with check (...)` — UPDATE should match for consistency. | New migration that drops and re-creates each UPDATE policy with both clauses: `using (id = auth.uid()) with check (id = auth.uid())`. Safe and behaviorally equivalent today; defensive for future changes. | fixed |
| db-005 | med | infra/supabase/migrations/20260401000000_init.sql:25-31,84-98,120-122 ; 20260501000000_purchases_v2.sql:57-59,105-107,128-130,154-156 | Every RLS policy in the schema calls `auth.uid()` directly. Supabase's Performance Advisor recommends `(select auth.uid())` to allow Postgres to cache the function result for the duration of the query — bare `auth.uid()` is re-evaluated **per row**. On small tables (current state) the impact is negligible; on `cosmetic_unlocks` / `achievement_progress` once they grow to thousands of rows per account, the per-row STABLE function call becomes measurable. See [Supabase RLS performance docs](https://supabase.com/docs/guides/database/postgres/row-level-security#use-functions-in-policies). | New migration that rewrites the affected policies to use `(select auth.uid())` instead of `auth.uid()`.
