# ClawBoy — Export Compliance Rationale

*Document type: Internal legal reference*
*Effective: May 10, 2026*
*Reviewed by: Kirby Walls, Sunday Softworks*

---

## Summary

ClawBoy declares `ITSAppUsesNonExemptEncryption = false` in `app.json` (`ios.infoPlist`). This document explains why that declaration is correct and demonstrates that all cryptographic usage in the app qualifies for exemption under U.S. Export Administration Regulations (EAR).

---

## Applicable regulation

The relevant exemption is **EAR 15 CFR §740.17(b)(1) — Encryption Commensurate with Generally Available Products** (License Exception ENC). This exemption covers encryption that is:

1. Generally available to the public,
2. Implemented by a standards body, or
3. Used exclusively for authentication, digital signature, or data integrity verification (not confidentiality).

Apple requires the `ITSAppUsesNonExemptEncryption` key to be `false` when all encryption falls within the §740.17(b)(1) exemption and no export documentation (Annual Self-Classification Report, CCATS) is needed.

## Cryptographic inventory

### 1. TLS / HTTPS (wss://, https://)

| Property | Value |
|---|---|
| Algorithm | TLS 1.2/1.3 (AES-GCM, ChaCha20-Poly1305, ECDHE) |
| Implementation | iOS system networking stack (Apple-provided) |
| Key exchange | Handled by the OS — not app-level code |
| Exemption | §740.17(b)(1): generally available, OS-provided, standard protocol |

All outbound connections from the app use TLS:
- Gateway WebSocket connections default to `wss://` (app warns loudly before accepting `ws://`).
- Supabase API calls use `https://`.
- Expo OTA update checks use `https://`.
- Cloudflare feedback Worker uses `https://`.
- RevenueCat SDK uses `https://`.

No app-level code implements or wraps TLS.

### 2. Ed25519 — Device identity and gateway authentication

| Property | Value |
|---|---|
| Algorithm | Ed25519 (Curve25519, EdDSA) |
| Implementation | `@noble/ed25519` (JS, pure TypeScript, no native) |
| Fallback | `expo-crypto` Web Crypto API (Hermes, available Expo SDK 50+) |
| Key material | Ed25519 private key stored in iOS Keychain (`expo-secure-store`) |
| Export from device | Never — only the signed challenge response (64-byte signature) leaves the device |
| Purpose | Gateway authentication challenge-response signing only |
| Exemption | §740.17(b)(1): authentication and digital signature use only; does not provide confidentiality |

The device keypair is generated once on first launch. The private key never leaves the device. Only signatures are transmitted to the user's own gateway. This is textbook "authentication use" exemption.

### 3. AES-256-GCM — Local chat cache encryption

| Property | Value |
|---|---|
| Algorithm | AES-256-GCM (authenticated encryption) |
| Implementation | `@noble/ciphers` (JS, pure TypeScript) |
| Key derivation | 256-bit random key generated by `expo-crypto` `getRandomBytesAsync` |
| Key storage | iOS Keychain (`expo-secure-store`) |
| Data encrypted | Recent message cache (~200 messages per server profile), stored in AsyncStorage |
| Purpose | Protecting locally cached data at rest |
| Exemption | §740.17(b)(1): generally available, standard algorithm, used solely for local data protection — not a user-facing confidentiality service offered to others |

This is equivalent to the "standard encryption" used by thousands of iOS apps to protect on-device cache and preferences. It does not constitute an "encryption product" under EAR — the app encrypts its own locally cached data for the user who runs it, not as a service provided to third parties.

### 4. TOFU SPKI Certificate Pinning

| Property | Value |
|---|---|
| Algorithm | SHA-256 hash of gateway certificate SubjectPublicKeyInfo (SPKI) |
| Purpose | Trust-on-first-use TLS certificate integrity verification |
| Exemption | §740.17(b)(1): data integrity, not confidentiality; hash function (SHA-256) not considered encryption under EAR |

SPKI pinning stores a SHA-256 fingerprint of the remote server's public key. SHA-256 is a one-way hash function, not an encryption algorithm under EAR.

---

## Non-exempt encryption: None identified

No encryption is present in the app that would require a CCATS filing or Annual Self-Classification Report:

- No VPN functionality.
- No key exchange protocol implemented at the app layer (all handled by OS TLS).
- No end-to-end encryption of user data transmitted between parties.
- No encryption exported to third parties as a service.
- No proprietary or non-standard encryption algorithm.

---

## Conclusion

All cryptographic operations in ClawBoy fall within EAR §740.17(b)(1):

| Crypto use | Exemption basis |
|---|---|
| TLS (wss://, https://) | OS-provided, standard protocol, publicly available |
| Ed25519 device signing | Authentication use only, no confidentiality |
| AES-256-GCM cache | Standard algorithm, local data protection, not offered as a service |
| SHA-256 SPKI pinning | Hash function, data integrity, not encryption under EAR |

**`ITSAppUsesNonExemptEncryption = false` is the correct and accurate declaration.**

When App Store Connect prompts "Does your app use encryption beyond what Apple provides?", answer **No** and select "Exempt" for all applicable encryption categories.
