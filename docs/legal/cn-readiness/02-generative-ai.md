# Blocker 2 — Generative AI Interim Measures + Algorithm Filing

*Status: Deep issue — requires architectural + legal decisions*
*Last updated: May 10, 2026*

---

## What is this?

Two overlapping regulatory frameworks apply to ClawBoy's core functionality in China:

### A. Generative AI Service Management Interim Measures (生成式人工智能服务管理暂行办法)

Effective August 15, 2023. Issued jointly by the Cyberspace Administration of China (CAC / 国家互联网信息办公室), MIIT, MPS, MCA, SAMR, and NRTA.

**Scope:** Any entity providing generative AI services to the public in mainland China. "Generative AI service" includes text, images, audio, video, code, or other content generated by AI models.

**Key obligations:**
1. The model must pass a CAC security assessment before public launch.
2. Services must implement content filtering to prevent "七不准" (seven prohibited content categories): illegal content, disinformation, hate speech, violence, pornography, undermining national unity, content violating socialist core values.
3. Training data must meet provenance and copyright standards.
4. Real-name user registration required.
5. Service provider (not the end user) is responsible for content moderation.

**The ClawBoy problem:** ClawBoy connects to a user-configured gateway running arbitrary AI models. From a PRC regulator's perspective, ClawBoy is providing "generative AI services" regardless of whether the model runs on our servers or the user's. The user's gateway is not separately licensed.

### B. Algorithm Recommendation Management Provisions (互联网信息服务算法推荐管理规定)

Effective March 1, 2022. Requires algorithm providers to file with CAC/MIIT.

**Scope:** Any internet information service using algorithms to generate, recommend, or filter content shown to users.

**Key obligations:**
1. File algorithm details with MIIT Algorithm Filing System (互联网信息服务算法备案系统).
2. Label AI-generated content as AI-generated.
3. Provide users a path to opt out of algorithmic recommendations.

**The ClawBoy problem:** The gateway's model selection and inference algorithm is not under our control, and neither the user's gateway operator nor we have filed with MIIT.

## Viable paths for ClawBoy

### Path A: Partner with a registered domestic gateway operator

The cleanest path. An OpenClaw-compatible gateway provider based in PRC that:
1. Uses only CAC-registered models
2. Holds its own algorithm filing
3. Operates content filtering compliant with the Interim Measures
4. Is willing to offer API-compatible access to ClawBoy CN users

For the CN build, the onboarding flow presents only this partner gateway (no custom URL entry). The partner is the legally responsible generative AI service provider; ClawBoy is the thin client.

**Pros:** We are not the AI service provider — the partner is. Compliant quickly.
**Cons:** Dependency on partner availability, pricing, model quality. Users lose the "bring your own gateway" value prop in China.

### Path B: Whitelist domestic models on a self-operated PRC gateway

Sunday Softworks operates or licenses an OpenClaw gateway in mainland China, configured only with CAC-registered model APIs. File the algorithm with MIIT.

**Pros:** More control, our gateway, can eventually expand model list as more models register.
**Cons:** Requires PRC entity, PRC-region server, MIIT algorithm filing, content moderation implementation, ongoing compliance. Substantial ongoing cost.

### Path C: Restrict user-provided gateways to domestic models (trust-based, unverifiable)

Allow custom gateway URLs but only permit model names matching a domestic allowlist in the CN build. Gate this via `gatewayPolicy.allowedModelPatterns` already defined in `src/types/index.ts`.

**Problem:** We cannot verify the gateway actually uses those models. A user can name any model anything. This is cosmetic compliance, not real compliance, and would likely fail a regulator audit.

**Verdict:** Not viable as a standalone path.

## Architectural guardrail: GatewayPolicy

The `GatewayPolicy` type is already defined in `src/types/index.ts` as part of this plan's code changes:

```typescript
export interface GatewayPolicy {
  allowedModelPatterns?: string[];
  requireServerSideContentFilter?: boolean;
  region?: 'global' | 'cn';
}
```

In the `'cn'` build:
- `ALLOW_CUSTOM_GATEWAY_URLS = false` (feature flag) → hides URL input in `AddServerSheet.tsx`
- `ServerProfile.gatewayPolicy.region = 'cn'` on all CN-build profiles
- `ServerProfile.gatewayPolicy.allowedModelPatterns` carries the domestic model allowlist
- Model selector (`useModels.tsx`) filters displayed models against `allowedModelPatterns`

This seam exists now. The decision of *which* partner / model list to fill it with is deferred to after the entity decision.

## CAC Algorithm Filing

When ready to file:

**Portal:** 互联网信息服务算法备案系统 — [https://beian.cac.gov.cn](https://beian.cac.gov.cn)

**Filing requirements include:**
- Service provider information (PRC entity)
- Algorithm description: type (generative), use case, training data sources
- Content safety measures implemented
- User protection measures (real-name, opt-out, labeling)
- Sample algorithm outputs for review

**Timeline:** Approximately 30 business days for initial review. Amendments required if CAC requests changes.

---

## Content labeling requirement

All AI-generated content shown to PRC users must be labeled as AI-generated. This is a UI requirement. For the CN build, an "AI生成" watermark or disclaimer should appear on assistant message bubbles. The current `MessageBubble.tsx` does not have this — it must be added for any CN build.
